M — MODEL THE SYSTEM
Requirements & Constraints
1. System Purpose
The website is the digital flagship for The Atelier Collective, showcasing five specialized studios while enabling commerce, bookings, and brand storytelling.
It must feel:
•	Luxurious
•	Editorial
•	Minimal
•	Trustworthy
•	Timeless
________________________________________
2. User Types
User	Description
Guest Visitor	Browses studios, views services, brand story
Client	Books appointments, views wig inventory, makes inquiries
Admin / Staff	Manages inventory, bookings, content, availability
________________________________________
3. Functional Requirements
Core (Umbrella Brand)
•	Landing page with atelier storytelling
•	Brand philosophy & founders story
•	Studio navigation (5 Ateliers)
•	Contact & inquiry forms
•	Instagram / social embedding
•	SEO-optimized editorial pages
The Hair Atelier (Priority Studio)
•	Wig catalog (view stock, images, price, status)
•	Wig detail pages
•	Braiding appointment booking
•	Availability calendar
•	Booking confirmation notifications
Other Studios
•	Service listings per atelier
•	Gallery/portfolio
•	Inquiry / booking request forms (phase 1)
•	Upgradeable to full booking later
Admin
•	CMS for content (pages, images, text)
•	Wig inventory management
•	Booking management
•	Availability control
________________________________________
4. Non-Functional Requirements
Category	Target
Availability	99.9%
Performance	< 2s page load
Security	HTTPS, RBAC, secure auth
Scalability	Handle seasonal booking spikes
Maintainability	Modular, studio-based
SEO	Core Web Vitals optimized
Aesthetics	High-end editorial design
________________________________________
5. Constraints
•	Budget: Lean–mid (startup luxury brand)
•	Team: Small dev team / solo engineer friendly
•	Timeline: MVP first, expand later
•	Traffic: Low initially, burst during promos
________________________________________
6. Success Metrics
•	Conversion rate (bookings)
•	Bounce rate < 40%
•	Booking completion time < 2 minutes
•	Zero downtime during promotions

T — TRADEOFFS
Technology Decisions
1. Frontend
Chosen: Next.js
✔ SEO
✔ Elegant routing
✔ Performance
✖ Slight learning curve
________________________________________
2. Backend
Chosen: Node.js (API routes or NestJS later)
✔ Fast to ship
✔ Large ecosystem
✖ Needs structure discipline
________________________________________
3. Database
Chosen: PostgreSQL
✔ Strong relations (bookings, services)
✔ Transactions
✖ Slightly heavier than NoSQL
________________________________________
4. API Protocol
Chosen: REST
✔ Simple
✔ Admin-friendly
✖ Less flexible than GraphQL
________________________________________
5. Deployment
Chosen: Serverless-first
✔ Low ops
✔ Scales automatically
✖ Vendor lock-in risk
